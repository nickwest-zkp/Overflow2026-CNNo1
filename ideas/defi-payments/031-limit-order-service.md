# 限价单服务 / Limit Order Service

> 在 Sui 上实现限价单功能，价格触达时通过 PTB 自动执行兑换

## 解决什么问题

去中心化交易所大多只支持市价单，用户无法设置"价格到 X 时自动买入/卖出"的限价单。这意味着用户需要持续盯盘才能在理想价格成交，极不方便。限价单服务在 Sui 上实现完整的限价单功能，用户设定目标价格后，系统在价格触达时通过 PTB 自动执行兑换，让链上交易体验接近中心化交易所。

## 核心功能

- 限价买单/卖单：设置目标价格和数量，价格触达时自动执行
- 条件限价单：支持"价格到达 A 后，如果回撤到 B 则买入"等高级条件
- 部分成交处理：大额订单可分批成交，减少滑点
- 订单有效期：支持 GTC（一直有效）、GTD（到日失效）、IOC（立即成交或取消）

## 使用的 Sui 原语

| 原语 | 用途 |
|------|------|
| Move 对象模型 | LimitOrder 作为独立 Object，存储订单参数和状态 |
| PTB | 原子化执行"价格检查 → DEX 兑换 → 更新订单状态" |
| Coin 标准 | 处理订单中的代币锁定和兑换 |
| 动态字段 | 存储订单历史、部分成交记录和手续费 |
| 预言机集成 | 通过 Pyth 获取实时价格触发订单 |

## 实现方案

### 智能合约 (Move)
- `order_factory` 模块：创建限价单，锁定代币到订单对象中
- `order_book` 模块：管理所有活跃限价单的注册和匹配
- `execution_engine` 模块：价格触达时通过 DEX 执行兑换
- `partial_fill` 模块：处理部分成交和剩余订单管理
- 关键数据结构：`LimitOrder { owner: address, pay_token: Type, buy_token: Type, amount: u64, limit_price: u64, filled: u64, status: u8, expiry: u64 }`

### 前端 / 后端
- 技术栈：React + @mysten/dapp-kit + Sui TypeScript SDK
- 关键页面：Place Order（下单）、Active Orders（活跃订单）、Order History（历史订单）、Price Alert（价格提醒）
- 后端 Keeper 监控价格并触发订单执行

## 技术架构

用户创建限价单时通过 PTB 将代币锁定到 LimitOrder 对象中，设置目标价格和有效期。Keeper 服务持续监控预言机价格与活跃订单的匹配，当市价触达限价时通过 PTB 执行：从订单中取出锁定代币 → 通过 DEX 兑换为目标代币 → 将兑换结果转入用户钱包。部分成交时更新已成交数量，剩余部分继续等待。订单到期时未成交部分自动退还。
